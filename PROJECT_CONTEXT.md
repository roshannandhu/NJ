## 1. What this app is

An **offline-first desktop application** for NJ India Trading (a roofing-products
seller in Kozhikode, Kerala). A salesperson builds **quotations** from a product
catalogue, optionally issues **warranty certificates**, prints/shares them as
PDFs, and relies on an aggressive **auto-backup & recovery** subsystem so data is
never lost on a single shop PC.

- **Runs as a native Windows window** (pywebview / WebView2), no browser, no
  console. The FastAPI backend runs in-process on `127.0.0.1` on a random free
  port; the built React SPA is served by that same backend (same origin).
- **No cloud server, no multi-user, no login server.** All data lives in one
  local SQLite file. The only "auth" is an optional local PIN.
- **Stdlib-only backend where possible** (the backup engine has zero third-party
  deps) so the bundled Python stays small and installs clean on a seller's PC.

### Tech stack
| Layer | Tech |
|---|---|
| Frontend | React 19 + Vite 8, plain inline styles + a few `.css` files, `lucide-react` icons |
| PDF | **client-side** `jsPDF` + `html2canvas` (bundled, offline). WeasyPrint was dropped. |

## 3. Data storage & environment configuration

### DATA_DIR resolution (`database.py`)
- `NJ_DATA_DIR` env var → holds **all writable data** (DB + `uploads/` + cloud
  token files + `launch.log`).
- In **dev**: defaults to the `backend/` folder.
- When **installed** (under read-only Program Files), `run_app.py` sets
  `NJ_DATA_DIR=%LOCALAPPDATA%\NJ India Data` — kept separate from the program
  folder so reinstalling never deletes data.
- `NJ_DB_PATH` overrides the DB file path (used by tests to point at a throwaway
  copy). Default: `DATA_DIR/nj_india.db`.

### SQLite configuration
- **WAL journal mode** + `synchronous=NORMAL` set on every connection (crash/power-loss safe, lets reads proceed during backup writes).
- `check_same_thread=False` (backup/scheduler daemon threads share the engine), `timeout=30` (wait out brief write overlaps instead of "database is locked").

### Lightweight migrations (no Alembic)
`ensure_columns()` runs on every startup and `ALTER TABLE ... ADD COLUMN` for any
missing column. Currently tracks `updated_at` + `version` on `quotations` and
`warranty_certificates`. **To add a column to an existing table, add it to
`_EXPECTED_COLUMNS` in `database.py`** — do not rely on `create_all()` alone (it
won't alter pre-existing tables).

### Other env / files
- `cloud_config.json` — OAuth client ids/secrets the user pastes in (per machine, gitignored).
- `cloud_tokens.json` — OAuth tokens, **DPAPI-encrypted** per-user when on Windows, plain JSON otherwise (gitignored).
- `launch.log` — desktop launcher diagnostics.

## 4. Database schema (`models.py`)

Only **5 tables**. Note the deliberate design: rich/structured data is stored as
a **JSON blob in a `data` TEXT column**; a few scalar columns are denormalised
out of that blob purely for querying/sorting/conflict-resolution.

### `app_config` (single row, id=1)
| col | type | notes |
|---|---|---|
| id | Integer PK (=1) | always one row |
| data | Text (JSON) | **the entire catalogue**: `company`, `settings`, `brands[]`, `classes[]`, `varieties[]`, `warranties[]` |

### `backup_state` (single row, id=1)
| col | type | notes |
|---|---|---|
| id | Integer PK (=1) | |
| data | Text (JSON) | backup targets, schedule, logs, recovery log, change journal, cached health score — see `default_state()` |

### `quotations`
| col | type | notes |
|---|---|---|
| id | String PK | e.g. `NJ-Q-123456` (minted on the frontend) |
| customer_name | String | denormalised from `data.customer.name` |
| grand_total | Float | denormalised |
| date | String | display date `dd/mm/yyyy` |
| created_at | DateTime | utcnow |
| updated_at | DateTime | bumped every write (`onupdate`) |
| version | Integer | edit counter, mirrored into `data` blob → drives conflict resolution |
| data | Text (JSON) | the **full quotation object** (items, customer, tax/discount, bank, terms, notes, etc.) |

### `warranty_certificates`
| col | type | notes |
|---|---|---|
| id | String PK | deterministic, e.g. `NJ-W-123456-docke` |
| quotation_id | String | **FK-by-convention** to `quotations.id` (no DB-level FK; enforced in app code) |
| customer_name | String | denormalised |
| date | String | |
| created_at / updated_at / version | | same pattern as quotations |
| data | Text (JSON) | the **full certificate object** (template snapshot, certData, customer, items) |

### Relationships & integrity
- **`warranty_certificates.quotation_id → quotations.id`** is the only
  relationship. It is **NOT a database foreign key**; integrity is enforced in
  application code:
  - **No-orphan rule (backend `warranties.py`):** saving a warranty *requires* a
    non-empty `quotationId` AND that the parent quotation exists, else HTTP 400.
  - **Cascade delete (backend `quotations.py`):** deleting a quotation (single or
    "clear all") also deletes its warranty certificates in the same transaction.
- There are **no FK constraints, no indexes** beyond the primary keys.

## 6. The backup / verification / recovery engine (`backup_service.py`)

This is the most important and most complex backend module (~1800 lines). It is
**dependency-free** and **never raises into request handlers** (everything is
best-effort).

### Backup "set" = 3–4 files sharing a timestamp stem `nj_backup_YYYYMMDD_HHMMSS`:
- `.db` — atomic SQLite copy (via SQLite online backup API, safe during writes)
- `.json` — full payload (config + quotations + warranties), `kind` tagged
- `.uploads.zip` — images (only when catalogue changed, or forced on manual)
- `.manifest.json` — sha256s, row counts, verify results, per-target status

### Background daemons (started in `main.py` lifespan)
1. **Startup backup** (`run_startup_backup`) — one snapshot per launch (skipped if a backup ran < 10 min ago, to avoid dev-reload spam).
2. **Daily scheduler** (`start_scheduler`) — wakes every 30 min; backs up if `interval_days` elapsed; also auto-analyses a freshly-synced destination.
3. **Event worker** (`start_event_worker`) — debounced: any data change calls `notify_change()`, which marks dirty; after `EVENT_QUIET_SECONDS` (8s) of calm it takes one coalesced `"event"` backup.
4. **Verification loop** (`start_verification_loop`) — every `verify_interval_minutes` runs `run_verification()`: scans newest backup vs live DB, and if `auto_recover_enabled`, **imports everything missing** and **resolves clear conflicts**.
On shutdown: daemons stop + a final `"shutdown"` backup.

### Destinations (`targets`)
`local` (Documents\NJ India Backups by default), `usb`, `gdrive`, `onedrive`.
- Folder targets: validated by `_target_dir_ok` — **must be an absolute drive path or UNC share** (a bare name like `192.168.1.11` is rejected; it used to create a junk relative folder).
- Cloud targets: handled by `cloud_backup.py` (real OAuth upload), not folder copies.
- NAS / Network / Dropbox were **removed** (NAS/Network never networked; Dropbox unused). `_merge_defaults` strips them from any old saved state.

### Restore semantics (critical — see Business Rules)
- **`merge` (default, non-destructive):** history upserted by id (nothing deleted); config lists unioned by id; `company`/`settings` shallow-merged. Implemented in `_apply_config` + `restore_from_payload`.
- **`replace` (destructive):** wipes history then inserts payload; config becomes exactly the payload. **Always** takes a `pre_restore_*` snapshot first.
- **`restore_catalog_payload`** touches ONLY config — never history (and vice-versa). This separation is intentional and must be preserved.

### Intelligent recovery (additive, never deletes)
- `analyze()` — diff backup vs live by record hash → missing / conflict / local-only, including config items and images. Caches a summary. **No writes.**
- `recover()` — restores ONLY selected records/files; adds missing, overwrites only ids in an explicit overwrite list. Snapshots first.
- `auto_recover_missing()` — imports all "missing" from a scan.
- `resolve_conflicts()` — for records in both that differ, the winner is decided by `(version, updatedAt)` tuple; if the backup is strictly newer it **archives** the live copy (to an `archive_*` folder) then overwrites; otherwise keeps live.
- `compute_health_score()` — 0–100 score from 5 weighted factors (backup recency 30, verification recency 20, destinations reachable 20, integrity 15, completeness 15). Cached in state.

### Cloud backup (`cloud_backup.py`)
- OAuth **Authorization-Code + PKCE**, loopback redirect, system browser.
- Google Drive (`drive.file` scope, needs client_id **and** secret — "Desktop app" client) and OneDrive (Microsoft Graph, client_id only).
- Tokens DPAPI-encrypted at rest. Uploads into a "NJ India Backups" folder; rotates to `keep`.
- User must register their own free developer-console client id(s) and paste them in Settings.

## 7. Core business rules (extracted from actual code)

These are enforced in code today. **Do not break them.**

### Quotations
- **B-Q1 — Upsert by id, never duplicate.** `POST /api/quotations` upserts. The
  frontend keeps `activeQuotationId` for the current draft so "regenerate"
  UPDATES the same record (same number & date) instead of inserting a copy
  (`Checkout.finalizeGeneration`, `AppContext.loadQuotationForEdit`).
- **B-Q2 — Manager name is mandatory.** Checkout blocks generation without it; it
  is persisted on the quotation and remembered in `localStorage` (`nj_last_manager`).
- **B-Q3 — Customer name is mandatory** to generate.
- **B-Q4 — Pricing math (mirrored in `Checkout.jsx` and `QuotationDocument.recomputeTotals`):**
  - `actualPrice` = immutable master/Settings price captured at add-to-cart time.
  - `price` = effective (chargeable) price; lowering it below `actualPrice` creates an **"offer"** (the difference becomes "savings"); never mutate `actualPrice`.
  - `discountAmount` = percent → `round(subtotal*value)/100`; fixed → `min(value, subtotal)`.
  - `taxableAmount = subtotal − discount`; `taxAmount = round(taxable*taxRate)/100`; `grandTotal = taxable + tax`.
  - Tax/discount/installation/bank are **per-quotation overrides**, seeded from global `settings` but stored on the quotation (audit-safe).
- **B-Q5 — Bank snapshot.** The chosen bank object is copied onto the quotation so later Settings edits/removals don't alter an issued quotation.
- **B-Q6 — Editing an existing quotation keeps original number AND issue date;** only new quotations stamp today.
- **B-Q7 — Double-submit guard** (`isGenerating`) prevents duplicate records.

### Warranties
- **B-W1 — No orphan warranties, ever.** Backend rejects a warranty without an existing parent quotation. Deleting a quotation cascades to its warranties.
- **B-W2 — "Warranty Only" flow** mints a hidden `warrantyOnly: true` quotation (grandTotal 0) to back a standalone certificate, so the no-orphan rule holds. These hidden quotations **never appear in Quotation History or Dashboard**.
- **B-W3 — Deterministic certificate ids:** `<warrantyPrefix>-<qSuffix>-<sanitized template id>` (`warranty.js buildWarrantyCertsForQuotation`). Re-running for the same quotation reuses ids (upsert, no churn). One certificate **per distinct warranty template** the cart's product classes require.
- **B-W4 — Tools/accessories have no warranty.** Classes of `type:"tools"` have `warrantyId:null`, so they're naturally excluded from warranty generation and from brand grouping.
- **B-W5 — Certificates MIRROR the live template.** Logo, seal, signature, opening, sections, tables, durations all come from the *current* warranty template; editing a template instantly updates every certificate of that type. Per-customer data (`customer`, `certData`) lives on the certificate and is never overwritten. Only if the template was deleted does the certificate fall back to its frozen snapshot (`WarrantyDocument.jsx`). **Editing terms on a certificate edits the underlying template** (and persists to config).

### Catalogue
- **B-C1 — One config blob.** Brands → Classes → Varieties → Warranty templates, plus `company` & `settings`, all in `app_config.data`. `PUT /api/config` replaces it wholesale, so the frontend always sends the complete object (`AppContext.persistConfig`).
- **B-C2 — Brand layer auto-migration.** On load, if a catalogue has no `brands`, a default "NJ" brand is created and every class assigned to it (idempotent, persisted once) — `AppContext` + `_apply_config`.
- **B-C3 — Warranty template backfill.** On load, empty `sections`/`seriesTable` are backfilled from `DEFAULT_DATA`, but explicit user values (including `false` display flags) are preserved.

### Backup / data safety
- **B-D1 — merge is non-destructive; replace is destructive** and always snapshots first. Recovery/verification are **additive** (never delete).
- **B-D2 — Catalogue and history restores are isolated** (one can never delete the other).
- **B-D3 — Conflict resolution is version-based** (`version`, `updatedAt`); the live copy is archived before any overwrite.
- **B-D4 — Backups never block the app** and never raise into requests.

### App / security
- **B-S1 — Backend-offline guard.** If the backend is unreachable on load, the app shows a blocking red banner and the user is told NOT to edit (seed defaults must never be saved over real data) — `AppContext`.
- **B-S2 — Optional PIN lock** (`settings.pinEnabled`, `settings.pin`). Master override PIN `999999` always unlocks (`App.jsx`). PIN is plaintext in config; this is a soft lock, not real security.

## 9. Key workflows (end-to-end)

### Create a quotation
1. **Quotation Desk** — enter customer name (required); pick brand → class → variety, set colour/qty, **Add** (captures `actualPrice`).
2. (Desk sets `generateIntent`: `quote` / `both` / `warranty`.)
3. **Checkout** — adjust prices (offers), qty; toggle tax/discount/installation; pick bank; enter **manager name (required)**; **Finalize**.
4. `Checkout.finalizeGeneration` → `createQuotation(snapshot)` (upsert) → if `both`/`warranty`, `buildWarrantyCertsForQuotation` + `createWarranty` per cert.
5. Lands on **Quotation Document** (or **Warranty Document** for warranty-only). Inline-editable; Download/Word/Share.
6. Every save → backend `notify_change` → debounced event backup.

### Issue a warranty
- From a quotation ("Create Warranty" / intent `both`), or standalone ("Warranty Only" → hidden backing quotation).
- One cert per distinct template required by the cart's classes; deterministic ids; certs mirror the live template.

### Edit an existing record
- History/Dashboard → Edit → `loadQuotationForEdit` rehydrates cart/customer/id → Checkout → Finalize updates the same record (same id/date). Linked warranties re-sync without id churn (`syncCertToQuotation`).

### Backup & recovery
- Automatic: startup + daily + debounced-on-change + periodic verification with auto-recovery.
- Manual: Backup page "Back up now"; download full/catalogue/history/images ZIPs; restore (merge/replace); scan+recover specific records.

### Share a PDF (Windows)
- Frontend renders PDF bytes (jsPDF) → POST `/api/share-pdfs` → backend writes real files → `ShareHelper.exe` opens the native Share flyout (WebView2 has no Web Share API). Fallback: download.

## 12. Performance considerations

- **Full-table JSON deserialization:** list endpoints (`/api/quotations`,
  `/api/warranties`) and `build_payload()` load every row and `json.loads` each
  `data` blob. Fine for a single shop's volume; would not scale to tens of
  thousands of records. The denormalised columns exist but list endpoints don't
  use them (they return the parsed blob).
- **Backup writes copy the whole DB** each time (atomic copy + full JSON export).
  Debounced (8s quiet) + interval-gated so it's not per-keystroke, but a very
  large DB makes each backup heavier — hence the health-score size warnings
  (amber ≥50MB, red ≥200MB).
- **`recoveryScan`/`analyze`** hashes every record on both sides; O(n) per scan,
  run periodically by the verification loop.
- **html2canvas capture** at up to 3× DPI is CPU-heavy for long multi-page
  quotations (clamped to safe canvas limits). This is the slowest user-facing op.
- **No DB indexes** beyond PKs; ordering is by `created_at` (full scan + sort).
- Frontend keeps the entire history in `data` in memory and re-derives lists with
  `.filter`/`.map` on every render (no virtualization). Fine at expected scale.

## 13. Security risks / notes

Context: single-user desktop app bound to `127.0.0.1`. Risks are scoped accordingly.

- **No authentication on the API.** Any local process can call it. CORS is `*`.
  Acceptable for localhost desktop; do **not** expose this server on a network.
- **PIN is plaintext** in the config blob and has a hardcoded master override
  (`999999`). It's a UI lock, not a security boundary.
- **`restore-path` reads an arbitrary server-side file path** from the request
  body. Local-only, but it is unrestricted file read of any JSON-parseable file.
- **OAuth tokens** are DPAPI-encrypted per-user on Windows (good) but fall back to
  **plain JSON** off-Windows. `cloud_config.json`/`cloud_tokens.json` are
  gitignored — never commit them.
- **`usb_drives` shells out to `wmic`** (deprecated on newer Windows; may return
  empty). `share.py` shells out to `csc.exe`/`ShareHelper.exe`.
- **Uploaded filenames** are prefixed with epoch ms and `Path(...).name` (basename
  only), which neutralises path traversal; content type isn't validated.
- No rate limiting, no CSRF protection (not meaningful for a localhost single-user app, but note it if this ever becomes networked).

## 14. Known invariants a change must respect

- The **SPA mount must stay last** in `main.py` (after all `/api` routers).
- New columns go in **`_EXPECTED_COLUMNS`** (`database.py`), not just the model.
- All frontend↔backend calls go through **`api.js`** (or `share.js` for share).
- `PUT /api/config` expects the **complete** catalogue object.
- Quotation/warranty **ids are minted on the frontend** and are the upsert key.
- `version`/`updatedAt` must keep being **mirrored into the JSON blob** on every
  write (recovery depends on it).
- Backup code must remain **stdlib-only** and **never raise** into requests.
